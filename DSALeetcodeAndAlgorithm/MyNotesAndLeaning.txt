Exception rules in inheritance

Throw a subclass of the exception

9. ClassA a = new ClassB(); — Polymorphism

If a subclass overrides a method from superclass, the subclass version is called at runtime.

If the subclass doesn’t override a method, the superclass method is used.

This is called runtime polymorphism / dynamic method dispatch.

Example:

class A {
    void show() { System.out.println("A"); }
}
class B extends A {
    void show() { System.out.println("B"); }
    void extra() { System.out.println("Extra"); }
}

public class Test {
    public static void main(String[] args) {
        A a = new B();
        a.show();      // prints "B"
        // a.extra();  // ❌ Compile-time error, reference type A doesn't have extra()
    }
}


Key: Reference type decides what methods can be called (compile-time), object type decides which implementation runs (runtime).

Interfaces and default methods use this concept — the default method runs if not overridden.

10️. Objects and Heap

When you create an object using new, object and its instance data are stored in the heap.

References point to that heap object. Modifying the object through any reference affects the same object.

Example:

p1.name = "Alice";

Person p2 = p1;  // both references point to same object
p2.name = "Bob";

System.out.println(p1.name);  // prints "Bob"


Both p1 and p2 point to the same heap object.

14. equals() and hashCode()

Always override both for proper behavior in Hash-based collections.

Example:

class Employee {
    int id;
    String name;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Employee)) return false;
        Employee e = (Employee)o;
        return id == e.id && name.equals(e.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id, name);
    }
}

public class TestHashCodeEquals {
    public static void main(String[] args) {
        Employee e1 = new Employee(1, "Alice");
        Employee e2 = new Employee(1, "Alice");
        Employee e3 = new Employee(2, "Bob");

        System.out.println(e1.equals(e2)); // true
        System.out.println(e1.equals(e3)); // false

        Map<Employee, String> map = new HashMap<>();
        map.put(e1, "Developer");
        map.put(e2, "Tester"); // overwrites because hashCode and equals same
        map.put(e3, "Manager");

        System.out.println(map.size()); // 2
        System.out.println(map.get(e1)); // Tester
    }
}
Key Points:

equals() checks logical equality (content)

hashCode() must be consistent with equals

Hash-based collections (HashMap, HashSet) use hashCode first, then equals
Ensures HashMap or HashSet works correctly using both id and name.

16. Compile-time Polymorphism (Method Overloading)

Compile-time polymorphism happens when methods have the same name but different parameters.

The compiler decides which method to call based on reference type and parameters.

class Calculator {
    int add(int a, int b) {
        return a + b;
    }

    double add(double a, double b) {
        return a + b;
    }
}

public class Test {
    public static void main(String[] args) {
        Calculator calc = new Calculator();
        System.out.println(calc.add(5, 10));     // calls int version → 15
        System.out.println(calc.add(5.5, 2.5));  // calls double version → 8.0
    }
}


✅ Key: Decision is made at compile time → method signature decides which method is called.

17. Downcasting Example

Downcasting converts a superclass reference pointing to a subclass object into a subclass reference.

Must be safe — object must actually be the subclass.

class Animal {
    void sound() { System.out.println("Animal sound"); }
}

class Dog extends Animal {
    void sound() { System.out.println("Dog barks"); }
    void fetch() { System.out.println("Dog fetches"); }
}

public class TestDowncasting {
    public static void main(String[] args) {
        Animal a = new Dog(); // upcasting
        a.sound();            // Dog barks (runtime polymorphism)

        // a.fetch(); // ❌ Compile-time error (Animal reference doesn't know fetch)

        // Downcasting
        Dog d = (Dog) a; // safe because a actually refers to Dog
        d.fetch();       // Dog fetches
    }
}


✅ Key: Upcasting → automatic, downcasting → explicit with (Subclass)

19. volatile

Purpose: ensures visibility of changes to a variable across threads.

Without volatile, one thread may not see updates made by another thread.

class Counter {
    volatile int count = 0;
    
    void increment() {
        count++;
    }
}


Note: volatile does not guarantee atomicity. If multiple threads do count++, race conditions can occur.


Race Condition

A race condition happens when two or more threads access and modify the same variable at the same time, and the final result depends on the order in which threads execute.

Even if the variable is volatile, it does not make operations atomic — it only ensures visibility.

Example with Counter
class Counter {
    volatile int count = 0;

    void increment() {
        count++; // NOT atomic
    }
}

public class TestRace {
    public static void main(String[] args) throws InterruptedException {
        Counter counter = new Counter();

        Runnable task = () -> {
            for (int i = 0; i < 1000; i++) {
                counter.increment();
            }
        };

        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);

        t1.start();
        t2.start();

        t1.join();
        t2.join();

        System.out.println("Final count: " + counter.count);
    }
}

Expected vs Actual

Expected: 2000 (1000 increments by t1 + 1000 by t2)

Actual: Often less than 2000

Why?

count++ is actually 3 steps under the hood:

Read the value of count from memory

Add 1 to the value

Write the new value back to memory

If two threads do this at the same time, steps can interleave:

Thread 1 reads count = 5
Thread 2 reads count = 5
Thread 1 writes count = 6
Thread 2 writes count = 6   <- lost update!


This is the race condition.

✅ Key Points

volatile ensures all threads see the latest value.

volatile does NOT make increment (or other compound operations) atomic.

To avoid race conditions for count++, you can use:

synchronized method/block

AtomicInteger

AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet(); // atomic

with synchronized:
class Counter {
    int count = 0;

    // synchronized ensures only one thread can execute at a time
    synchronized void increment() {
        count++;
    }
}

public class TestSynchronized {

        t2.start();

}

ExecutorService Overview

ExecutorService is a higher-level replacement for manually creating Threads.

It manages a thread pool and handles task execution.

Key methods:

submit(Runnable/Callable) → submit tasks

shutdown() → stop accepting new tasks

shutdownNow() → attempt to stop running tasks

2️⃣ Common ExecutorService Types
a) Fixed Thread Pool

Executors.newFixedThreadPool(int nThreads)

Fixed number of threads, tasks wait in a queue if all threads are busy.

ExecutorService fixedPool = Executors.newFixedThreadPool(3);

for (int i = 1; i <= 5; i++) {
    int task = i;
    fixedPool.submit(() -> System.out.println("Task " + task + " executed by " + Thread.currentThread().getName()));
}

fixedPool.shutdown();


Example: 3 threads, 5 tasks → 3 run immediately, 2 wait.

b) Cached Thread Pool

Executors.newCachedThreadPool()

Creates new threads as needed, reuses idle threads.

Good for short-lived asynchronous tasks.

ExecutorService cachedPool = Executors.newCachedThreadPool();

for (int i = 1; i <= 5; i++) {
    int task = i;
    cachedPool.submit(() -> System.out.println("Task " + task + " executed by " + Thread.currentThread().getName()));
}

cachedPool.shutdown();


Threads die after 60 seconds of idleness.

c) Single Thread Executor

Executors.newSingleThreadExecutor()

Only one thread executes tasks sequentially.

Useful when tasks must run in order.

ExecutorService singlePool = Executors.newSingleThreadExecutor();

for (int i = 1; i <= 3; i++) {
    int task = i;
    singlePool.submit(() -> System.out.println("Task " + task + " executed by " + Thread.currentThread().getName()));
}

singlePool.shutdown();


Tasks run one by one, always same thread.

d) Scheduled Thread Pool

Executors.newScheduledThreadPool(int nThreads)

Executes tasks after a delay or periodically.

ScheduledExecutorService scheduledPool = Executors.newScheduledThreadPool(2);

scheduledPool.schedule(() -> System.out.println("Delayed task executed!"), 2, TimeUnit.SECONDS);
scheduledPool.scheduleAtFixedRate(() -> System.out.println("Periodic task!"), 0, 1, TimeUnit.SECONDS);

scheduledPool.shutdown();


Good for cron-like tasks or repeated jobs.

e) Virtual Thread Executor (Java 21+)

Executors.newVirtualThreadPerTaskExecutor()

Each task runs on a virtual thread (lightweight, managed by JVM).

Can create thousands of concurrent tasks without blocking threads.

ExecutorService virtualPool = Executors.newVirtualThreadPerTaskExecutor();

for (int i = 1; i <= 5; i++) {
    int task = i;
    virtualPool.submit(() -> System.out.println("Task " + task + " executed by " + Thread.currentThread()));
}

virtualPool.shutdown();


Threads are not OS threads, very lightweight, ideal for I/O-heavy apps.

20. Abstraction

Purpose: Hides implementation details, shows only what the object can do.

Achieved using:

Abstract classes → can have abstract + concrete methods

Interfaces → only abstract methods (default/static methods allowed in Java 8+)

Example:

abstract class Animal {
    abstract void sound(); // abstract method, implementation hidden
}

class Dog extends Animal {
    void sound() {
        System.out.println("Woof");
    }
}

Abstract Method

Definition: A method without a body (no implementation).

Declared with abstract keyword.

Example:

abstract class Animal {
    abstract void sound(); // abstract method → no body
}


Key point: Subclasses must provide implementation (unless the subclass is also abstract).

2️⃣ Concrete Method

Definition: A method with a body / implementation.

Can be in abstract or regular classes.

Example in abstract class:

abstract class Animal {
    abstract void sound();  // abstract method
    
    void sleep() {          // concrete method
        System.out.println("Sleeping...");
    }
}


sleep() is concrete because it has an implementation.

Subclasses inherit it automatically and don’t need to override it.

Example usage:

class Dog extends Animal {
    void sound() {
        System.out.println("Woof");
    }
}

public class Test {
    public static void main(String[] args) {
        Dog dog = new Dog();
        dog.sound(); // Woof
        dog.sleep(); // Sleeping...
    }
}


sound() → abstract, must override

sleep() → concrete, inherited

28. Internal Working of HashSet and HashMap:
HashMap put() behavior

Hashing: Key (e.g., "Roopa") → hash code → bucket index in underlying array.

Bucket: Each index holds a linked list (or tree if many collisions, Java 8+) of key-value pairs.

Case 1 — New key:

map.put("Roopa", 1); // hashed, bucket index created, entry added


Case 2 — Collision (same bucket, different key):

map.put("Apoor", 2); // same bucket → linked list of nodes created


Case 3 — Same key:

map.put("Roopa", 2); // key exists → value updated (overwrites previous)

2️⃣ putIfAbsent()

Adds only if key does not exist.

Does not overwrite existing values.

map.putIfAbsent("Roopa", 3); // key exists → value remains 2

3️⃣ computeIfAbsent()

If key does not exist, compute value and add.

If key exists → do nothing.

map.computeIfAbsent("Roopa", k -> 100); // key exists → value remains 2
map.computeIfAbsent("NewKey", k -> 50); // key absent → value 50 added

4️⃣ computeIfPresent()

If key exists, compute new value and update.

If key absent → do nothing.

map.computeIfPresent("Roopa", (k,v) -> v+10); // value updated to 12
map.computeIfPresent("AbsentKey", (k,v) -> 5); // key absent → nothing happens

5️⃣ HashSet behavior

Implemented using HashMap internally:

HashSet<String> set = new HashSet<>();
set.add("Roopa"); 


Internally → map.put("Roopa", PRESENT)

If you add("Roopa") again → checks contains("Roopa") → does not add duplicate.

6️⃣ Key points

HashMap: Key → hashed → bucket → linked list / tree → value stored

HashSet: Uses a HashMap internally; values ignored, only keys matter

put → adds or updates

putIfAbsent → adds only if absent

computeIfAbsent → computes only if absent

computeIfPresent → computes only if present



29. ClassLoader Basics

Purpose: Load .class files (bytecode) into the JVM at runtime.

Hierarchy:

ClassLoader	Loads
Bootstrap	Core Java classes (java.lang.*, rt.jar)
Extension (Platform)	ext or lib/ext classes
System / Application	Classes from your application’s classpath

Every class in JVM is loaded by some ClassLoader.

2️⃣ When main() is executed

Suppose you have:

public class TestClass {
    public static void main(String[] args) {
        System.out.println("Hello World");
    }
}

Step-by-Step Execution

JVM starts → loads bootstrap classes (like java.lang.Object, java.lang.String) using Bootstrap ClassLoader.

ClassLoader loads your class (TestClass) → System/Application ClassLoader reads TestClass.class from classpath.

main() method is found in the loaded class.

main() is public static void main(String[] args) → static, JVM can call without creating an object.

Memory allocation:

JVM creates a stack frame for main().

Any objects created inside main() go to heap.

Bytecode execution:

The interpreter reads bytecode line by line or JIT compiler converts hot code to native machine code for faster execution.

Program terminates:

JVM frees stack frames, GC may clean heap objects.

    TestClass loaded → main() found
            |
    Stack frame created → main() executed
            |
    Heap objects created as needed
            |
    JIT compiler optimizes hot code
            |
    Program ends → GC cleans heap

37. clone()

Purpose: Create a copy of an object.

Must implement Cloneable interface, otherwise CloneNotSupportedException is thrown.

class Employee implements Cloneable {
    int id;
    String name;

    Employee(int id, String name) {
        this.id = id;
        this.name = name;
    }

    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone();
    }
}

public class TestClone {
    public static void main(String[] args) throws CloneNotSupportedException {
        Employee e1 = new Employee(1, "Alice");
        Employee e2 = (Employee) e1.clone();
        System.out.println(e2.name); // Alice
    }
}


Shallow copy → primitive fields copied, object references are shared.

Deep copy → create new objects for reference fields.
Employee e1 = new Employee(1, "Alice");
Employee e2 = (Employee) e1.clone();
e2.name = "Roopa";
Employee implements Cloneable and uses super.clone() → this is a shallow copy.

Shallow copy rules:

Primitive fields (like int id) → copied → independent in e1 and e2.

Reference fields (like String name) → copied as references → both objects point to the same object.

But in this case, String is immutable in Java.

Step-by-step:
e1.name = "Alice" → String literal in the String pool

e2 = e1.clone() → e2.name points to same "Alice" literal

e2.name = "Roopa" → e2.name now points to a new String literal "Roopa"

Important: The name field now points to a different String object. Immutable strings make shallow copy behave almost like deep copy for this field.

Result:
System.out.println(e1.name); // Alice
System.out.println(e2.name); // Roopa
e1.name remains Alice because strings are immutable and assigning a new string to e2.name does not modify the original object.

⚠️ If name were mutable (like StringBuilder):
class Employee implements Cloneable {
    int id;
    StringBuilder name;

    Employee(int id, StringBuilder name) { this.id = id; this.name = name; }

    protected Object clone() throws CloneNotSupportedException { return super.clone(); }
}

Employee e1 = new Employee(1, new StringBuilder("Alice"));
Employee e2 = (Employee) e1.clone();

e2.name.append("X"); // modifies same object!
System.out.println(e1.name); // AliceX
Here, shallow copy means e1 and e2 share the same mutable object.

Deep Copy:
Goal: Create a completely independent copy of the object, including all reference fields.

Solution: Manually clone the references inside clone().

class Employee implements Cloneable {
    int id;
    StringBuilder name; // mutable

    Employee(int id, StringBuilder name) {
        this.id = id;
        this.name = name;
    }

    // Deep copy
    @Override
    protected Object clone() throws CloneNotSupportedException {
        Employee cloned = (Employee) super.clone(); // shallow copy first
        cloned.name = new StringBuilder(this.name.toString()); // clone mutable field
        return cloned;
    }
}

public class TestDeepCopy {
    public static void main(String[] args) throws CloneNotSupportedException {
        Employee e1 = new Employee(1, new StringBuilder("Alice"));
        Employee e2 = (Employee) e1.clone();

        e2.name.append("X");

        System.out.println(e1.name); // Alice (unchanged)
        System.out.println(e2.name); // AliceX
    }
}


✅ Result:

e1.name is not affected by changes in e2.name.

All mutable fields must be copied manually.
38. Memory leak -
What is a Memory Leak?

Definition: Memory leak occurs when objects are no longer needed by the program but cannot be garbage collected because some references are still held.

Effect: Over time, your program consumes more memory → can cause OutOfMemoryError.

2️⃣ How Memory Leak Happens in Java

Even though Java has Garbage Collection (GC), leaks can happen if references are unintentionally retained.

Common causes:

Static Collections holding objects

class LeakExample {
    static List<Employee> employeeList = new ArrayList<>();

    void addEmployee(Employee e) {
        employeeList.add(e); // never removed → memory leak
    }
}


Listeners / Callbacks not removed

button.addActionListener(listener);
// never removed → listener keeps reference alive


Caches without eviction

Map<Integer, Employee> cache = new HashMap<>();
// keeps growing → old objects never removed


Inner classes holding outer class reference

class Outer {
    class Inner { }
}
// Inner instance keeps reference to Outer → prevents GC


Unclosed resources (before Java 7)

File, stream, database connection not closed → memory and resource leak.

3️⃣ Example of a Simple Memory Leak
import java.util.ArrayList;
import java.util.List;

public class MemoryLeakDemo {
    static List<int[]> list = new ArrayList<>();

    public static void main(String[] args) {
        while (true) {
            list.add(new int[1000]); // objects kept in static list → never GC’d
        }
    }
}


Problem: Static list keeps growing → memory usage increases → eventually OutOfMemoryError.

4️⃣ How to Avoid Memory Leaks

Remove references when not needed (list.remove(obj)).

Use weak references for caches:

Map<Integer, WeakReference<Employee>> cache = new HashMap<>();


Deregister listeners when no longer needed.

Use try-with-resources for streams and connections:

try (FileInputStream fis = new FileInputStream("file.txt")) { ... }


Avoid unnecessary static fields holding objects.

5️⃣ Key Points for Interview

Java GC does not prevent memory leaks if references are retained.

Memory leak ≠ memory not freed — it’s memory unreachable but still referenced.

Can lead to OutOfMemoryError over time in long-running applications.

41. Fail Fast and Fail Safe:

Fail-Fast

Definition: Iterators that immediately throw ConcurrentModificationException if the collection is modified while iterating (except through the iterator itself).

Collections: ArrayList, HashMap, HashSet (most non-concurrent collections)

Example:

import java.util.ArrayList;
import java.util.Iterator;
import java.util.List;

public class FailFastExample {
    public static void main(String[] args) {
        List<String> list = new ArrayList<>();
        list.add("Alice");
        list.add("Bob");
        list.add("Charlie");

        Iterator<String> it = list.iterator();

        while (it.hasNext()) {
            System.out.println(it.next());
            list.add("David"); // modifying list directly
        }
    }
}


Output:

Alice
Exception in thread "main" java.util.ConcurrentModificationException


✅ Explanation:

Direct modification of list while using iterator → fail-fast behavior.

2️⃣ Fail-Safe

Definition: Iterators that do NOT throw ConcurrentModificationException if the collection is modified while iterating.

Collections: CopyOnWriteArrayList, ConcurrentHashMap

Mechanism: Fail-safe iterators iterate over a clone/copy of the collection.

Example with CopyOnWriteArrayList:

import java.util.Iterator;
import java.util.concurrent.CopyOnWriteArrayList;

public class FailSafeExample {
    public static void main(String[] args) {
        CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();
        list.add("Alice");
        list.add("Bob");
        list.add("Charlie");

        Iterator<String> it = list.iterator();

        while (it.hasNext()) {
            System.out.println(it.next());
            list.add("David"); // safe, no exception
        }

        System.out.println("Final list: " + list);
    }
}


Output:

Alice
Bob
Charlie
Final list: [Alice, Bob, Charlie, David, David, David]


✅ Explanation:

Iterator works on a snapshot of the list → modifications don’t affect the iteration.

This is fail-safe.

3️⃣ Key Differences Table
Feature	Fail-Fast	Fail-Safe
Throws Exception	✅ ConcurrentModificationException	❌
Iterates over	Original collection	Copy/snapshot of collection
Collections	ArrayList, HashMap, HashSet	CopyOnWriteArrayList, ConcurrentHashMap
Performance	Fast	Slower (copies made)
Use Case	Single-threaded	Multi-threaded safe iteration

Fail-fast → “fail immediately to detect concurrent modification”

Fail-safe → “safe to modify during iteration, usually slower”

🔴 43. Deadlock (VERY IMPORTANT INTERVIEW TOPIC)
What is Deadlock?

A situation where two or more threads are waiting forever for each other’s locks.

Simple Deadlock Example
class DeadlockExample {
    static final Object lock1 = new Object();
    static final Object lock2 = new Object();

    public static void main(String[] args) {

        Thread t1 = new Thread(() -> {
            synchronized (lock1) {
                System.out.println("Thread 1: Holding lock1");
                try { Thread.sleep(100); } catch (Exception e) {}
                synchronized (lock2) {
                    System.out.println("Thread 1: Holding lock2");
                }
            }
        });

        Thread t2 = new Thread(() -> {
            synchronized (lock2) {
                System.out.println("Thread 2: Holding lock2");
                try { Thread.sleep(100); } catch (Exception e) {}
                synchronized (lock1) {
                    System.out.println("Thread 2: Holding lock1");
                }
            }
        });

        t1.start();
        t2.start();
    }
}

What happens?

Thread-1 holds lock1, waits for lock2

Thread-2 holds lock2, waits for lock1

❌ Both wait forever → DEADLOCK

Deadlock Conditions (Coffman Conditions)

All 4 must be true:

Mutual Exclusion – only one thread can hold a lock

Hold and Wait – thread holds one lock and waits for another

No Preemption – lock can’t be forcefully taken

Circular Wait – circular dependency of locks
